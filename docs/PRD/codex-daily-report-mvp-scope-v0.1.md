## 1. 决策摘要

`v0.1.0 MVP` 不先建设完整的本地分析应用，而是交付一个可安装、可自动运行的 **Codex 日报生成器**：

```text
launchd 每日触发 / 用户手动执行
-> 启动一次性 CLI 进程
-> 只读扫描 Codex 本地日志
-> 识别工作区间、项目、任务和成果
-> 计算使用指标、评分证据和当日 L1-L4
-> 经授权执行标准 AI 分析，失败时降级为指标报告
-> 输出自包含 HTML 日报和版本化机器可读数据
-> 按需打开默认浏览器
-> 进程退出
```

MVP 只支持 Codex，只生成日报。它不包含常驻 daemon、localhost API、React SPA、SQLite 事实库、多工具、周报、月报或交互式纠错。

## 2. 为什么继续压缩

### 2.1 要验证的核心假设

首个版本只需要回答：

> 从真实 Codex 会话生成的每日“使用数据、工作成果、协作层级和教练建议”，是否足够准确、可信并有行动价值，使用户愿意持续查看？

在这个假设成立前，提前建设多工具接入、常驻服务、动态 Dashboard、复杂纠错和完整发布外壳，只能验证工程可实现性，不能证明日报有产品价值。

### 2.2 复杂度裂缝

上一版范围的主要复杂度集中在产品化外壳：

- 多工具带来字段口径、版本兼容和故障隔离矩阵。
- 常驻 Web 服务带来进程生命周期、端口、鉴权、CSRF、CSP 和升级问题。
- SQLite 与交互式纠错带来迁移、派生结果所有权、重算和事务一致性问题。
- 日周月报告和导出带来周期聚合、版本传播和多视图一致性问题。
- `.app` 分发带来签名、公证、sidecar、升级和卸载边界。

将首版改成一次性进程和静态报告后，上述复杂度大部分不再是 MVP 的前置条件，而 Codex 解析、任务识别、证据评分和日报价值仍能被完整验证。

### 2.3 范围演进记录

| 阶段 | 数据源 | 周期 | 产品形态 | 主要结论 |
| --- | --- | --- | --- | --- |
| PRD v0.2 | Codex、Claude Code、OpenCode、WorkBuddy、Qoder | 日、周、月 | 常驻本地 Web 应用 | 完整 V1 愿景，适合作为中期路线图 |
| 第一次压缩 | Codex、Claude Code | 日报 | daemon + Dashboard | 去掉多周期和三个数据源，但产品化成本仍较高 |
| 第二次压缩 | Codex | 日报 | daemon + Dashboard | 聚焦首条数据链路，但仍提前建设服务和动态前端 |
| 最终 MVP | Codex | 日报 | one-shot CLI + launchd + 静态 HTML | 直接验证日报内容和持续使用价值 |

## 3. MVP 包含范围

### 3.1 数据接入

- 自动发现受支持版本的 Codex 本地日志。
- 始终只读源日志，不修改时间戳、权限或内容。
- 只处理目标日报窗口需要的数据；不在首次运行时默认分析全部历史内容。
- 重复执行必须幂等，稳定事件标识不得包含可逆的原始内容或项目路径。
- 明确展示已发现、无数据、无权限、版本不支持和解析失败状态。

### 3.2 日报周期

- 日报窗口为本地时区的昨日 19:00 至今日 19:00，使用左闭右开区间。
- 支持每日 19:00 由 `launchd` 触发。
- 支持用户手动生成指定日报窗口。
- 设备休眠或关机错过触发后，在下次登录或运行时补生成缺失日报。
- 同一窗口重复生成不得产生重复统计；新结果替换该窗口的当前报告，同时保留生成时间和分析版本。

### 3.3 报告内容

HTML 日报按固定顺序展示：

1. **数据与层级**：Token、会话、消息、工具调用、Skill、Subagent、活跃时间、当日得分、L1-L4 和关键证据。
2. **工作成果**：工作区间、项目候选、任务名称、成果摘要、验证状态和置信度。
3. **教练建议**：最多 3 条，每条包含问题证据、下一次行动和验证方式。

使用强度本身不能提高评分。证据不足时显示“不可评分”或“证据不足”，不得用 0 或模型猜测替代缺失事实。

### 3.4 AI 分析

- 未授权或未配置模型时，仍生成完整的确定性指标和证据报告。
- 标准 AI 分析首次使用前必须明确授权，并展示发送的数据类型和目标 provider。
- MVP 不接收模型凭据；标准分析复用 Codex CLI 已有登录态，并使用 ephemeral 模式避免生成新的源会话。
- 发送内容采用最小分析包，排除明显 secret、大段代码、完整工具输出和无关历史。
- 模型失败不得阻塞日报生成；报告标记降级原因并保留可重试入口。

### 3.5 输出与本地运行

- 每个日报窗口生成一个自包含 HTML 文件，浏览器不依赖本地服务器或远程 CDN。
- 同时保存带 schema version 的报告 JSON；解析中间结果可使用 JSONL 和扫描 manifest。
- 提供简单的本地历史日报索引页，不提供跨日报聚合、筛选或交互式修改。
- 安装流程配置 CLI、`launchd` 和本地数据目录；卸载明确区分程序、派生报告、配置和不受影响的 Codex 源日志及登录态。
- 支持一键清除全部派生报告和分析数据，但不触碰 Codex 源日志。

### 3.6 最小质量门禁

- Codex 合成 fixture 的解析、窗口边界、稳定去重和幂等测试。
- 任务识别和评分 eval 按已批准规则运行。
- 隐私、secret scanning、恶意日志内容、路径边界和 HTML 注入测试。
- CLI 安装、手动生成、定时生成、补偿生成和卸载 smoke test。
- 构建、类型检查、lint 和测试在本地与 CI 中通过。

## 4. 不能继续削减的底线

以下能力共同构成 MVP 的价值闭环，任何一项被删除都需要重新定义 MVP：

1. 使用真实 Codex 日志，而不是只展示合成数据或 mockup。
2. 同时交付“数据、成果、建议”，而不是只有 Token 统计。
3. L1-L4、任务和建议带证据与置信度，不做不可解释的模型打分。
4. 每日自动运行并支持补偿，验证持续使用行为，而不是一次性 Demo。
5. 源日志只读、分析明确授权、凭据安全存储、派生数据可清除。
6. Codex 分析不可用时仍能生成确定性日报，避免模型能力成为单点故障。

## 5. 明确延期的内容

| 延期能力 | 原因 | 恢复条件 |
| --- | --- | --- |
| Claude Code | 第二个真实 adapter 会扩大兼容与口径矩阵，但不增加首个日报价值假设的验证强度 | Codex 日报内容通过 MVP 验收，且 UnifiedEvent 契约稳定 |
| OpenCode、WorkBuddy、Qoder | 当前分别存在版本数据、许可或能力限制 | 对应 spike 结论允许实现，且前一个 adapter 接入未要求重写核心契约 |
| 周报、月报和 7/30 天趋势 | 需要足够日报样本才有真实价值 | 至少积累 14 个有效日报，并出现明确的跨日复盘需求 |
| 常驻 daemon 和 localhost API | 静态日报不需要服务端交互，提前建设会引入额外安全面 | 出现实时刷新、交互式设置、跨报告查询或纠错需求 |
| React/Vite Dashboard | 自包含 HTML 足以验证信息结构和阅读价值 | 静态报告通过验证，且交互需求无法通过轻量 HTML 满足 |
| SQLite 事实库 | 单来源单周期可由源日志、JSONL、manifest 和报告 JSON 支撑 | 多来源、跨周期聚合、纠错重算或并发访问成为确定需求 |
| 深度分析 | 逐次预览、授权、取消和结果版本会扩大隐私与状态空间 | 标准日报稳定，且用户明确需要对指定会话进一步分析 |
| 人工纠错和完整重算 | 需要持久事实模型、优先级和事务边界 | 任务误判成为影响留存的主要问题，并有可重复的纠错模式 |
| Markdown/JSON 用户导出 | MVP 的用户交付物是 HTML；内部 JSON 是机器契约，不承诺外部兼容 | 出现迁移、分享或外部集成需求后定义正式导出契约 |
| `.app`、签名、公证、自动升级 | 与日报价值无关且发布工程成本高 | CLI 版本通过公开 alpha 验证，准备面向非开发者分发 |
| 钉钉日历、排行榜、团队能力 | 不属于个人本地日报的核心假设 | 作为独立产品决策重新评估，不从 MVP 默认外推 |

延期不代表否定。恢复能力必须由上表的真实信号触发，而不是因为它曾出现在旧路线图中。

## 6. 为扩展保留的最小契约

MVP 只保留已经有明确未来消费者的边界：

- `UnifiedEvent`：统一事件的版本化 schema，当前只有 Codex producer。
- `DailyReport`：指标、成果、评分、建议、完整度和分析版本的稳定输出 schema。
- `Analyzer`：确定性分析与可选模型分析的边界。
- `ReportRenderer`：同一 DailyReport 到 HTML 的投影边界。
- `SourceAdapter`：仅定义 Codex 实现实际需要的最小读取契约，不提前建设插件注册、动态加载或多工具配置系统。

JSON/JSONL 文件必须包含 schema version，未来 SQLite 或 Dashboard 通过显式 migration 导入。不得为了假设中的未来能力扩大 MVP schema、UI 或运行时。

## 7. MVP 验收与继续投资信号

### 7.1 工程验收

- 在干净且受支持的 macOS 账户完成安装、手动生成、定时生成、错过补偿和卸载。
- 连续 7 个有 Codex 活动的日报窗口中，至少 6 个无需人工修复即可成功生成。
- 重复运行不重复计数，源日志在验证前后内容、时间戳和权限不变。
- 未配置模型或模型失败时，指标版日报仍可阅读。
- 报告不包含凭据、未脱敏绝对路径、大段代码或完整原始会话。

### 7.2 分析验收

- Codex 任务名和项目归属的一致率达到已批准的 overall 80% 门槛；单工具 70% floor 在只有一个工具时不单独降低 overall 门槛。
- L3 至少有 3 个同时具备迭代与验证证据的任务。
- L4 至少有 5 个同时具备迭代与验证证据的任务，且至少 2 个任务具有复用资产证据。
- 每个得分、等级、风险和建议都能关联到结构化证据或明确标记证据不足。

### 7.3 产品继续信号

完成 7 个有效日报后再决定是否进入完整本地应用。继续投资至少需要同时出现：

- 用户能够在一分钟内理解当天主要数据、成果和协作层级。
- 多数任务名称和成果无需回看原始会话即可理解。
- 至少一类教练建议被认为可在下一次协作中执行和验证。
- 用户愿意继续自动生成，而不是只在首次体验时查看。

若这些信号不成立，优先调整日报内容、证据和评分，不增加数据源或产品外壳。

## 8. 对现有路线图和 Issues 的影响

### 8.1 MVP 关键路径

- **#6**：ADR 改为确定 one-shot CLI、`launchd`、静态 HTML、Codex ephemeral 分析、JSON/JSONL、路径安全和安装边界。
- **#7**：从本地 daemon + 五数据源 Dashboard 收缩为 CLI 启动、Codex 状态和静态报告外壳。
- **#8**：保留 Codex 只读、幂等导入，但按日报窗口和轻量中间数据实现。
- **#13-#17**：保留任务识别、日报、标准分析、评分和固定信息顺序；删除多周期、28 天趋势和动态下钻要求。
- **#24**：保留 MVP 所需的 CI、解析、隐私、secret 和安装 smoke 门禁。
- **#25**：收缩为 CLI、`launchd`、安装/卸载脚本和公开 alpha 验证；`.app`、签名、公证和自动升级延期。

- **#9-#12**：其他数据源延期。
- **#18-#21**：深度分析、完整纠错、周月报告、正式导出和精细删除延期；全量清除作为 MVP 安装生命周期的一部分保留。
- **#31-#36**：保持非阻断 follow-up，不阻塞日报 MVP。

## 9. 接受与拒绝的方案

### 接受

- 先验证 Codex 日报价值，再扩展为完整产品。
- one-shot CLI + `launchd`，每次生成后退出。
- 自包含 HTML 作为首个用户界面。
- JSON/JSONL + schema version 作为 MVP 中间与输出契约。
- 一个真实 Codex adapter 和最小扩展接口。

### 当前拒绝

- 为“以后可能需要”提前搭建通用多工具插件系统。
- 在没有交互需求前运行常驻 localhost 服务。
- 在没有跨周期、纠错或并发访问需求前引入 SQLite 事实库。
- 用多数据源数量、动态 Dashboard 完整度或安装包装替代日报价值验证。
- 因缩减范围而降低证据、隐私、幂等和失败降级要求。
