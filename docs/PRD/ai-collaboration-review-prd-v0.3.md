> 本期只解决一件事：把一个日报窗口内的大量 Codex 会话可靠地重建为任务、语义轮次和跨会话关系，再基于可核对证据生成成果、L1-L4 和教练建议。

## 1. 模块定位

本模块位于“原始日志解析”和“日报评分与展示”之间。它不直接追求更长的 AI 总结，而是先建立可信的任务事实层，产出任务关系图和任务证据包，驱动后续成果复盘、协作评分和教练建议。

核心判断是：`Session` 只是日志容器，不等于一个任务；消息数量不等于迭代轮次；同一项目和相邻时间只能作为关联线索，不能直接证明多个会话属于同一任务。

## 2. 当前问题与基线证据

### 2.1 当前问题

公开 Alpha 已证明安装、扫描、定时生成和静态 HTML 交付可运行，但真实日报的分析可信度不足：

- 系统内容可能被当作用户目标，导致任务标题出现系统指令或环境上下文。
- 同一项目下不同会话容易被时间规则合并，任务数被过度压缩。
- 任意多条消息或工具调用可能被当作“发生迭代”，无法区分有效调整和重复循环。
- 关键词出现可能被当作验证或资产沉淀，无法证明测试真的执行、产物真的形成。
- AI 当前接收的是全日报截断消息，而不是每个任务完整且均衡的证据包；开启 AI 也只能润色错误任务，不能修复任务边界。
- 数据完整度、任务置信度与 L1-L4 没有形成足够强的门禁。

## 3. 功能目标

### 3.1 用户价值

- 用户能在一分钟内看懂当天真正完成了哪些任务，而不是阅读会话标题集合。
- 用户能看到一个任务经历了几轮有效推进、是否跨会话继续、是否由 Subagent 分支协作。
- 用户能区分事实、推断和证据不足，不会被虚高的 L1-L4 误导。
- 当结论存疑时，用户可以从任务卡下钻到必要证据，而不必阅读整天原始日志。

### 3.2 AI 价值

- AI 每次分析一个边界清晰的任务证据包，而不是吞入全日报前若干条消息。
- AI 负责语义判断和摘要；Token、时间、会话标识和工具状态等确定性事实仍由本地规则提供。
- AI 只有在证据不足时才按预算回读指定过程片段，并记录回读原因。

### 3.3 成功标准

1. 在至少 20 个本地人工标注的真实 Codex 任务上，任务边界整体一致率不低于 80%；真实内容和逐条结果不进入 Git。
2. 系统内容、环境上下文和工具输出被误用为任务标题的比例为 0。
3. 跨会话关联以准确优先：错误合并率不高于 10%，错误拆分率不高于 20%。
4. 在至少 30 个已标注意义轮次上，“是否发生有效迭代”的判断一致率不低于 80%。
5. 验证、Subagent 运行和资产沉淀证据的准确率不低于 90%；未知状态不得按已发生处理。
6. 人工活跃时间在单日报窗口内不超过窗口时长；并行 Agent 时间单独展示且不重复计入。
7. 数据或任务证据不足时不输出无条件 L3/L4；报告明确展示降级原因。
8. AI 分析包覆盖全部可评分任务，不因全局截断只分析当天前部会话。
9. 同一份源日志在相同统计口径下产生一致的会话、任务、消息和 Token 事实指标。
10. 主观洞察全部带证据锚点和置信度；没有证据的观察不进入日报结论或评分。

### 3.4 非目标

- 不新增 Claude Code、OpenCode、WorkBuddy 或 Qoder。
- 不新增周报、月报、跨日趋势或长期任务记忆。
- 不建设 daemon、动态 Dashboard、SQLite 或交互式人工纠错。
- 不改变 19:00 至次日 19:00 的日报窗口。
- 不用更大模型替代任务边界和证据规则。
- 不要求保存完整会话副本，不把真实日志、代码或对话写入仓库。
- 不在本期自动修改 Prompt、创建 Skill 或操作用户项目。
- 不把所有主观洞察都变成评分维度，也不为填满模板而强行生成无关结论。

## 4. 用户场景

### 场景 A：查看可信的当日任务

用户打开日报，先看到数据质量和使用事实，再看到按任务组织的成果。每个任务展示关联会话数、有效轮次、最终产出、验证状态和置信度。系统内容不会成为任务标题。

### 场景 B：同一任务跨会话继续

用户上午提出需求，下午在新会话中继续修复，期间还启动多个 Subagent。系统以明确延续语句、父子关系和共同交付物为主要证据，将会话组织为一个任务及其分支；仅因同项目或时间接近不自动合并。

### 场景 C：一个会话包含多个任务

用户在长会话中先完成发布，再转向另一个无关问题。系统识别目标切换，将其拆成两个任务；共享会话不代表共享任务。

### 场景 D：低置信度时渐进回读

系统先使用目标、结果、验证和产物等主干信息分析任务。只有当任务边界、成果或评分证据不足时，才回读相关工具结果或中间消息；仍无法确认时标记“证据不足”，不猜测。

## 5. 人与 AI 双轨协作

| 阶段 | 人工动作 | AI 动作 | 系统反馈 | 边界 |
| --- | --- | --- | --- | --- |
| 日报触发 | 手动运行或等待 19:00 调度 | 无 | 展示扫描和生成状态 | 不重复启动同一窗口生成 |
| 事实整理 | 无需逐条操作 | 不参与 Token、时间和来源事实计算 | 展示解析完整度和未知事件 | 原始日志只读，未知不猜测 |
| 任务重建 | 无 | 识别目标变化、轮次语义和跨会话候选关系 | 输出任务、关系、轮次和置信度 | 低置信关系不自动成为确定事实 |
| 渐进分析 | 首次使用 AI 前确认授权 | 先分析任务证据包，必要时请求受限回读 | 展示完整、部分成功或降级 | 不发送整天日志，不绕过内容预算 |
| 评分与报告 | 阅读证据并决定是否采信建议 | 总结成果并生成最多 3 条建议 | 展示评分依据和缺失证据 | 评分只消费结构化证据，不消费无锚点结论 |
| 后续验证 | 在下一次协作中实践建议 | 后续日报观察是否出现目标证据 | 建议只标记“观察到改善”或“尚未观察到” | 不凭模型措辞宣告完成 |

## 6. 核心对象与业务规则

### 6.1 Session：会话容器

- Session 由 Codex 源标识确定，是事件来源容器。
- 一个 Session 可以承载多个 Task；一个 Task 可以关联多个 Session。
- 系统指令、`AGENTS.md`、环境上下文和开发者指令属于上下文，不属于真实用户目标。
- 没有真实用户目标的 Subagent Session 优先关联父任务；无法关联时保留为“未归类执行”，不独立生成猜测任务。

### 6.2 Session Analysis：会话事实画像与洞察画像

每个 Session 先形成一份内部会话分析卡，再参与任务重建。日报不平铺展示全部会话卡，而是在任务卡和当日洞察中按需引用。

#### 6.2.1 事实画像

事实画像由本地确定性逻辑计算，不依赖 AI。固定方向包括：

| 方向 | 事实指标示例 | 口径要求 |
| --- | --- | --- |
| 会话规模 | 会话数、主会话数、Subagent 会话数、有效会话数 | 系统注入或空会话不计为有效工作会话 |
| 任务承载 | 每会话任务数、每任务关联会话数、跨会话任务数 | Session 与 Task 分开统计 |
| 消息交互 | 用户消息、AI 消息、有效交互轮数、消息总数 | 系统和工具事件不混入用户消息 |
| Token 消耗 | 输入、缓存输入、输出、推理和总量 | 缓存输入单列，不伪装成新增输入 |
| 时间投入 | 会话跨度、人工活跃时间、Agent 并行时间 | 跨度不等于活跃时间，重叠区间去重 |
| 工具与 Agent | 工具调用、唯一工具、Subagent 运行及状态 | 运行次数与状态交互事件分开 |
| 结果与验证 | 有成果会话、已验证任务、失败和未完成状态 | 未观察到不等于失败或为零 |
| 项目分布 | 项目数、各项目任务和消耗占比 | 项目归属不足时标记候选 |

日报在适用时至少提供以下派生指标：

- 平均每个有效会话的消息、Token、活跃时间和任务数。
- 平均每个任务的会话、语义轮次、消息、Token、工具调用和活跃时间。
- 平均每个有效轮次的模型调用和 Token 消耗。
- 验证任务占比、跨会话任务占比、Subagent 参与任务占比。

平均值必须与样本数、中位数共同展示；样本足够时增加 P90。少量极端长会话不得仅通过平均值代表典型使用。

Token 归因遵循保守原则：若源日志只提供模型调用级 usage，则展示“每次模型调用”或“每个有效轮次”的 Token，不展示无法证明的“每条消息 Token”。只有存在稳定消息级归因时，才显示消息平均消耗。

#### 6.2.2 洞察画像

洞察画像由 AI 基于会话骨架和证据生成，分为固定分析方向与内容驱动洞察。

固定分析方向用于保持不同会话之间的可比较性：

| 方向 | 核心问题 | 典型观察 |
| --- | --- | --- |
| 目标与约束 | 用户是否说明目标、背景、边界和完成标准 | 目标清楚、约束缺失、过程中目标漂移 |
| 推进与轮次 | 是否根据新信息调整并逐步收敛 | 有效迭代、机械重试、过早结束 |
| 工具与 Agent 编排 | 工具、Skill 和 Subagent 是否与任务复杂度匹配 | 合理并行、过度编排、该委派但未委派 |
| 结果与验证 | 是否形成明确成果并验证 | 已验证、仅口头完成、验证失败后修复 |
| 效率与风险 | 消耗是否产生新信息、决策或产出 | 上下文膨胀、重复读取、无效循环 |
| 资产与延续 | 是否形成可复用产物或为后续任务建立上下文 | 方法文档、脚本、Skill、未沉淀的重复工作 |

AI 还可以根据实际内容补充开放洞察，例如：

- Prompt 表达方式和值得复用的提问结构。
- 调试、研究、决策、设计或交付中出现的独特协作模式。
- 用户纠偏方式、验收意识或跨会话延续习惯。
- 某类任务中反复出现的上下文缺口、工具摩擦或高价值工作流。
- 固定方向未覆盖但对理解当日工作明显重要的异常或亮点。

开放洞察不要求每个会话都有。单个会话最多保留 2 条，且必须满足“与任务结果相关、有证据、能形成解释或行动”至少两项；纯内容摘要、人格判断和无法验证的心理推断不属于洞察。

#### 6.2.3 事实与洞察的关系

- 事实回答“发生了什么、多少、何时”；洞察回答“为什么值得注意、反映了什么协作模式”。
- 洞察可以引用事实，但不能修改事实。例如 Token 高只能触发进一步分析，不能直接得出“过度思考”。
- 单次会话洞察默认是观察，不直接成为稳定能力结论；跨多个任务重复出现后，才可进入日报教练建议。
- 固定洞察方向不要求强行给分。不适用或证据不足时显示“不适用”或“证据不足”。
- 内容驱动洞察只有能映射到正式五维证据时才进入评分，否则作为独立观察展示。

### 6.3 Task：用户要完成的工作

一个可评分任务至少需要：可识别目标，以及结果、明确未完成状态或可核对的推进结果之一。

任务边界按以下优先级判断：

1. 用户明确提出新目标、切换问题或继续既有任务。
2. 明确的父子会话、Subagent 或任务引用关系。
3. 相同 Issue、PR、文件产物、测试目标或明确交付物。
4. 目标与成果在语义上连续。
5. 相同项目目录和时间接近仅作为弱证据。

错误合并比错误拆分更影响成果和评分，因此证据冲突时默认拆分或标记候选关联。

### 6.4 Task Relation：跨会话关系

跨会话关系只使用以下产品语义：

- **继续**：新会话继续同一目标或处理同一未完成结果。
- **委派**：主会话将子目标交给 Subagent，子会话属于任务分支。
- **共享交付物**：不同会话共同修改或验证同一明确交付物，但是否同一任务仍需目标连续性支持。
- **无关**：仅项目或时间接近，没有目标和交付物证据。

每条自动关系必须保留类型、证据锚点和置信度。关系图只在当前日报窗口及必要的相邻上下文内重建；跨日报持久任务关系延期。

### 6.5 Semantic Round：语义轮次

语义轮次描述一次有意义的推进，而不是一条消息或一次工具调用。

一轮通常包含：

```text
触发：新目标 / 新反馈 / 新观察
→ 尝试：方案、工具行动或委派
→ 结果：工具结果、用户反馈或产物
→ 决策：继续、调整、验证、完成或停止
```

计入新轮次的条件至少满足一项：

- 根据新证据修改假设、方案或实现。
- 用户对结果纠偏后重新执行。
- 验证失败后定位原因并采取不同动作。
- 子任务结果返回主任务后改变下一步决策。

以下不单独计为有效轮次：

- 同一尝试内连续调用多个工具。
- 没有新信息的重复读取、重复报错或机械重试。
- 状态轮询、Subagent 交互消息和系统上下文注入。
- 仅重复总结既有内容。

无新信息的重复过程可以形成“无效循环”风险信号，但不能提高迭代得分。

### 6.6 Evidence Packet：任务证据包

每个任务证据包按三层组织：

| 层级 | 默认处理 | 内容 |
| --- | --- | --- |
| 核心主干 | 始终进入任务分析 | 真实用户目标、关键约束、用户反馈、最终结果、验证结论、明确产物 |
| 过程证据 | 结构化压缩后进入 | 语义轮次、错误与调整、工具行动摘要、Subagent 生命周期、Skill 使用 |
| 原始详情 | 默认只保留引用 | 大段工具输出、重复历史、完整代码、系统指令、环境上下文 |

证据包必须按任务分配内容预算，并保证每个可评分任务至少获得主干覆盖。不得以“全局前 N 条消息”代替任务级覆盖。

## 7. 渐进分析流程

### 7.1 总体流程

1. **事实扫描**：解析、去重、统计来源完整度，保留未知事件。
2. **来源分类**：区分真实用户内容、AI 回复、系统上下文、工具行动、工具结果和 Agent 生命周期。
3. **会话骨架**：为每个会话生成事实画像，提取目标候选、结果候选、产物和关键时间点，并生成洞察候选。
4. **任务重建**：识别会话内任务切换，建立跨会话继续、委派和共享交付物关系。
5. **轮次重建**：把消息和工具事件压缩为有因果关系的语义轮次。
6. **证据包分析**：AI 按任务总结名称、成果、验证状态和协作模式。
7. **按需回读**：只有置信度不足或证据冲突时，读取相关原始详情；超过预算则降级。
8. **评分与报告**：先通过数据质量门禁，再生成五维得分、L1-L4 和教练建议。

可编辑图：[任务重建与渐进分析流程](../diagrams/task-reconstruction-progressive-analysis-flow.drawio)

### 7.2 AI 可以决定的内容

- 任务名称和目标摘要。
- 会话内是否发生真实目标切换。
- 多条证据是否构成同一语义轮次。
- 成果摘要、协作模式和可执行建议。
- 低置信候选关系的语义判断。

### 7.3 AI 不可以覆盖的内容

- Token、时间戳、源会话 ID 和工具事件状态。
- 已确认的父子会话关系和实际测试退出状态。
- 数据完整度、未知事件和被跳过记录。
- 未观察到的文件、Git、测试或 Subagent 完成事实。
- 用户未授权的原始内容范围。

## 8. 评分与 L1-L4 门禁

### 8.1 五个维度保持不变

- 目标表达。
- 协作编排。
- 迭代意识。
- 验证闭环。
- 资产沉淀。

本期不更换维度，重点修正每个维度消费的证据。

### 8.2 证据规则

- 消息多、工具多、Token 多或耗时长不构成正向证据。
- 迭代证据必须来自至少一次“新信息 → 调整 → 新结果”的语义轮次。
- 验证证据必须包含可识别的验证动作和结果；只出现“test”“验证”等文本不成立。
- Subagent 指标按唯一运行及其 started/completed/failed/interrupted 状态统计，交互事件不重复计数。
- 资产证据必须指向真实形成或更新的可复用产物；只讨论 Skill、文档或模板不成立。
- AI 结论必须引用证据包中的锚点；无锚点结论不进入评分。

### 8.3 置信度与降级

| 置信度 | 产品处理 | 是否进入 L3/L4 门槛 |
| --- | --- | --- |
| 高：0.80-1.00 | 作为确定任务或证据展示 | 是 |
| 中：0.60-0.79 | 标记“候选”，允许展示和人工核对 | 仅有第二类独立证据时进入 |
| 低：低于 0.60 | 保留为未归类或证据不足 | 否 |

以下任一情况发生时，不得输出无条件 L3/L4：

- 任务边界整体证据不足。
- L3/L4 所需迭代、验证或资产证据包含低置信项。
- 相关事件类型解析失败，导致关键状态未知。
- AI 分析仅覆盖部分可评分任务。

报告可以继续展示确定性指标，但等级显示“证据不足”或降至满足全部门槛的最高等级，并说明原因。

## 9. 日报信息结构

### 9.1 数据与质量

- 数据完整度、分析覆盖率和降级原因置于评分之前。
- Token 展示输入、缓存输入、输出和推理分项；缓存 Token 不与非缓存输入混为“新增工作量”。
- 会话数展示源会话；任务数展示重建任务，二者并列但不暗示一一对应。
- Subagent 展示唯一运行数及完成、失败、中断状态，不展示事件总数为“调用次数”。
- 人工活跃时间和并行 Agent 时间分开显示。
- 总量旁展示适用的均值、中位数和样本数；样本充分时展示 P90，避免极端会话扭曲典型水平。
- “每消息 Token”只有在日志支持消息级归因时展示，否则使用“每次模型调用 Token”或“每有效轮次 Token”。

### 9.2 会话与任务分析

- 日报先展示会话事实总览：有效会话、任务承载、消息、Token、活跃时间、工具和 Agent 分布。
- 会话洞察不单独堆成大量卡片，而是聚合为当日最重要的 3-5 个模式，并关联到对应任务。
- 用户展开任务卡时，可以查看关联会话的事实摘要、语义轮次和最多 2 条内容驱动洞察。
- 事实与洞察使用不同视觉标签，避免用户把 AI 推断误认为日志事实。

### 9.3 工作成果任务卡

每个任务至少展示：

- 任务名、项目候选、开始与结束时间。
- 关联会话数、有效轮次数和 Subagent 分支数。
- 目标、成果、验证状态和可复用产物。
- 任务边界置信度、分析状态和证据锚点。
- 无效循环或未验证完成等风险信号。

静态 HTML 可使用原生折叠区域展示轮次摘要和证据，不引入客户端脚本或本地服务。

### 9.4 教练建议

- 最多 3 条，优先选择影响最大且证据最充分的问题。
- 每条包含：观察、证据、下一次行动和验证方式。
- 不基于单纯 Token、消息数或工具次数提出“减少使用”建议。
- 任务重建置信度不足时，不生成针对个人能力的强结论。

## 10. AI 状态反馈与人工接管

| 状态 | 用户可见反馈 | 系统行为 |
| --- | --- | --- |
| 整理事实 | 正在扫描与分类会话事件 | 只计算确定性事实 |
| 重建任务 | 正在识别任务、关系和轮次 | 生成候选关系和置信度 |
| 分析任务 | 显示已分析任务数 / 总任务数 | 按任务证据包处理 |
| 回读详情 | 标明回读原因和受影响任务 | 只回读必要片段，不扩大到整天日志 |
| 部分成功 | 展示未覆盖任务和不可评分维度 | 保留可用事实并降级评分 |
| 证据不足 | 展示缺失证据类型 | 不猜测，不输出虚高等级 |
| 完成 | 展示分析覆盖率、版本和报告 revision | 发布同一窗口的新 revision |

人工确认边界保持现有规则：首次启用 AI 分析需要明确授权；授权范围发生实质变化时重新确认。用户不需要为每个自动任务证据包逐次确认。

## 11. 模块拆解与输入输出

| 模块 | 产品职责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| 事件分类 | 区分用户、系统、工具和 Agent 事件 | Codex 统一事件 | 带来源语义的事件流 |
| 会话骨架 | 生成事实画像，提取目标、结果、产物和关键节点 | 分类事件 | 会话事实画像、摘要骨架和洞察候选 |
| 任务重建 | 拆分会话内任务并关联跨会话关系 | 会话骨架、确定性关系 | 任务关系图、边界置信度 |
| 轮次重建 | 识别有效推进、重复循环和验证闭环 | 任务事件 | 语义轮次和风险候选 |
| 证据包 | 分层压缩并控制每个任务的分析预算 | 任务、轮次、原始引用 | 任务证据包、回读请求 |
| AI 分析 | 生成任务名、成果和语义判断 | 任务证据包 | 带锚点结论和置信度 |
| 评分门禁 | 校验覆盖率、证据质量和等级门槛 | 事实、任务、AI 结论 | 五维得分、L1-L4 或降级原因 |
| 日报渲染 | 按数据、成果、建议顺序展示 | 结构化日报 | 自包含静态 HTML |

## 12. 异常与边界

| 情况 | 产品处理 |
| --- | --- |
| 会话只有系统上下文，没有真实用户目标 | 不生成独立任务；尝试关联父任务，否则标记未归类 |
| 同一项目存在多个并发会话 | 默认独立，除非有继续、委派或共同交付物证据 |
| 一个长会话发生明确目标切换 | 拆为多个任务，共享会话引用 |
| 任务跨越日报窗口 | 窗口外相邻内容只作理解上下文，不计入本期指标和活跃时间 |
| Subagent 启动后无完成事件 | 状态显示未知或未完成，不推定成功 |
| 工具反复失败且动作未改变 | 标记重复循环，不计入有效迭代轮次 |
| 只观察到测试关键词，没有执行结果 | 验证状态为未观察到或尝试过，不标记已验证 |
| 只讨论了 Skill / 文档 / 模板 | 不标记资产沉淀，除非存在真实产物证据 |
| AI 分析超时或仅完成部分任务 | 发布部分报告，显示覆盖率，未覆盖任务不参与语义评分 |
| 内容预算不足 | 优先保留每个任务主干，减少过程详情；仍不足则降级而非丢弃尾部任务 |
| 数据状态为 partial | 评估缺失是否影响任务和评分证据，逐维降级，不因 partial 字样一律清零 |

## 13. 验收标准

### 13.1 会话事实与洞察

- [ ] 会话数、消息数、模型调用、Token 和时间指标在相同输入与口径下可重复计算。
- [ ] 平均每会话、每任务和每轮次指标同时展示分母；适用时同时展示中位数。
- [ ] 无法稳定归因到消息的 Token 不展示为“每条消息 Token”。
- [ ] 缓存输入、输出和推理 Token 不与普通输入混为单一工作量结论。
- [ ] 每条主观洞察包含方向、结论、证据锚点和置信度，并明确标记为 AI 推断。
- [ ] 固定方向不适用时允许为空，不为填满卡片强行生成结论。
- [ ] 单个会话最多保留 2 条内容驱动洞察，纯摘要或人格判断被拒绝。
- [ ] 单次会话洞察不会在没有跨任务证据时直接升级为稳定能力结论。

### 13.2 任务与关系

- [ ] `AGENTS.md`、环境上下文、开发者指令和工具输出不会成为任务标题。
- [ ] 一个会话中的两个明确无关目标可被拆成两个任务。
- [ ] 同一项目的并发独立会话不会仅因 20 分钟内有事件而合并。
- [ ] 明确父子关系的 Subagent 会话可关联到主任务，并保留独立执行状态。
- [ ] 明确继续同一 Issue、PR 或交付物的跨会话任务可被关联。
- [ ] 不确定关系展示候选和置信度，不静默强合并。

### 13.3 语义轮次

- [ ] 连续多个工具调用但没有新反馈只计为同一轮尝试。
- [ ] 用户纠偏后采用新方案可识别为新一轮。
- [ ] 测试失败后修改实现并再次测试可形成迭代与验证闭环。
- [ ] 相同错误和相同动作的机械重试被标记为重复循环，不提高迭代分。
- [ ] Subagent 状态轮询和交互消息不增加有效轮次数。

### 13.4 渐进分析

- [ ] 每个可评分任务都有独立证据包并获得核心主干覆盖。
- [ ] 系统不会只取全日报前若干条消息作为全部任务的共同输入。
- [ ] 只有低置信度或证据冲突任务触发原始详情回读。
- [ ] 回读内容受单任务和单日报预算限制，失败时明确降级。
- [ ] AI 输出中的任务、成果、验证和建议均可追溯到证据锚点。

### 13.5 指标与评分

- [ ] 人工活跃时间对重叠区间去重，单日报不超过 1,440 分钟。
- [ ] 并行 Agent 时间不与人工活跃时间直接相加。
- [ ] Subagent 按唯一运行和状态统计，不把 started、interacted、interrupted 混为调用数。
- [ ] 验证和资产沉淀不再使用关键词命中作为充分条件。
- [ ] 中低置信任务不会单独把日报推到 L3/L4。
- [ ] 部分分析或关键证据未知时，等级和维度显示明确降级原因。

### 13.6 隐私与兼容

- [ ] 源日志保持只读，验证前后内容、权限和修改时间不变。
- [ ] 真实会话、代码、Prompt、回复和私人路径不进入 Git、Issue、PR、CI 或公开报告。
- [ ] 未授权或 AI 不可用时仍生成确定性数据和候选任务报告。
- [ ] 静态 HTML 保持无客户端 JavaScript、无远程资源和严格文本转义。
- [ ] 同一源数据、解析版本和分析配置可重建同一日报 revision 的事实结果。

## 14. 已确认实施决策

1. 20 个真实任务和 30 个语义轮次的本地标注集由产品所有者完成主标注；确定性规则和合成 fixture 由自动测试复核。真实内容和逐条结果不进入 Git。
2. 跨会话关联以最终任务分组准确性为主要验收指标，关系边准确性作为诊断指标。
3. 中置信证据只有获得第二类独立证据时才可进入 L3/L4；确定性的测试结果、产物变化和 Agent 生命周期优先于文本语义判断。
4. 静态 HTML 每个任务最多展示 5 个关键轮次，其余只展示数量和证据索引。
