> 本期只解决一件事：将本机多种 AI 编程工具的使用日志转换为可核对的数据看板、工作成果复盘和 AI 协作改进建议。

## 1. 模块定位

AI 协作复盘台是独立、开源、Local-first 的本地 Web 应用。它从本机 AI 工具日志中还原“用了多少”“做了什么”和“协作质量如何”，生成日、周、月报告，帮助用户形成可验证、可迭代、可复用的 AI 协作习惯。

产品首页的信息优先级固定为：

1. 当前使用数据和协作层级。
2. 当期任务、项目与产出复盘。
3. AI 教练建议和下一步改进行动。

## 2. 用户问题与产品原则

### 2.1 用户问题

- AI 使用分散在多个工具中，缺少统一的频次、Token、会话、工具调用和活跃时间视图。
- 原始会话可以证明过程，却不能直接回答当天做了什么、投入在哪些项目、产出了什么。
- Token 多、Subagent 多、Skill 多或思考时间长，不必然代表高水平协作。
- 单次会话没有自然形成日、周、月尺度的成长趋势和改进闭环。

### 2.2 产品原则

1. **事实优先**：先展示数据和证据，再展示推断和建议。
2. **强度不等于能力**：Token、会话数、Skill 和 Subagent 使用量不直接加分。
3. **结论可下钻**：任务、产出、分数和风险信号都保留来源引用、置信度和纠错入口。
4. **数据由用户控制**：原始日志留在本机；语义分析只通过用户配置的模型 API 完成。
5. **用户纠错优先**：人工合并、拆分、重命名和排除会触发重算，且不会被 AI 自动覆盖。
6. **未知不等于零**：数据源不提供某项数据时显示“不可用”，不得按 0 计算。

## 3. 功能目标

### 3.1 用户价值

- 用一个看板查看五类 AI 工具的当期使用数据。
- 每天 19:00 后确认过去 24 小时的任务、项目、工作区间和关键产出。
- 了解当日协作得分和近 28 天 L1-L4 成熟度，并查看评分证据。
- 获得数量受控、可执行、可在下一周期验证的改进建议。

### 3.2 AI 价值

- 将异构日志转换为统一事件，识别工作区间、项目和任务。
- 从消息、工具调用、验证行为和结果中提取任务名、产出和协作模式。
- 基于任务复杂度、行为证据、结果证据和置信度生成五维评价。
- 识别无效循环、上下文膨胀、工具滥用、过度编排和未验证完成等风险。

### 3.3 V1 成功标准

1. 五个目标工具均有独立数据源状态，能区分已导入、未发现、无权限、版本不支持和部分数据。
2. 重复扫描不重复计数；手动重算不生成冲突报告。
3. 每个工具至少使用 20 个、总计至少 100 个经人工标注的任务组抽样验收；整体任务名和项目归属识别一致率不低于 80%，且单个工具不低于 70%。
4. 所有评分、等级和风险信号都能查看证据和置信度。
5. 用户纠错后，相关汇总、得分和报告可以重新生成。
6. 日、周、月报告均可自动生成、手动重算，并导出 Markdown 和 JSON。

### 3.4 非目标

- V1 不正式支持 Windows 和 Linux，但适配器接口不得与 macOS 路径绑定。
- V1 不建设中心账号、云端数据库、排行榜或团队管理后台。
- V1 不写入钉钉日历，不接入 Temporal，不提供 Webhook；这些进入 V1.1。
- V1 不以单一 Token 数量衡量生产力，不对用户做公开排名。
- V1 不自动修改 Prompt、创建 Skill、启动 Subagent 或操作代码仓库。
- V1 不上传匿名遥测、用户指标、报告、原始会话或真实项目内容。

## 4. 关键用户场景

### 场景 A：19:00 查看当日报告

用户打开本地看板，先看到过去 24 小时的使用数据、今日得分和近 28 天成熟度；随后查看任务、项目、活跃时段和产出；最后选择一条改进建议作为下一周期的实践目标。

### 场景 B：从分数下钻并纠正误判

用户点击某个评分维度或风险信号，查看对应的会话片段、工具调用摘要和结果证据。若任务归属或工作区间不正确，可重命名、改项目、合并、拆分或排除，然后重算报告。

### 场景 C：对指定会话执行深度分析

用户对指定日期、任务或会话选择深度分析。系统先展示拟发送内容类型、预计 Token、目标模型和隐私提示；用户确认后才发送更完整上下文。结果保存在本地，用于补充标准分析。

## 5. 人与 AI 双轨协作

| 阶段 | 人工动作 | AI 动作 | 系统反馈 | 边界 |
| --- | --- | --- | --- | --- |
| 数据接入 | 授予日志访问权限，必要时指定路径 | 无 | 展示检测、权限和导入状态 | 未发现数据不得显示为 0 |
| 本地整理 | 可手动发起扫描 | 识别工作区间、项目候选和任务边界 | 展示进度、新增事件和异常 | 原始日志只读；推断保留置信度 |
| 标准分析 | 配置自有 API 和模型 | 对最小分析包生成任务、产出、评价和建议 | 展示分析、部分成功、失败与重试 | 过滤大段代码、重复上下文和完整工具输出 |
| 深度分析 | 选择范围并逐次确认 | 使用更完整上下文重新分析 | 发送前展示范围、预计 Token 和模型 | 未确认不得发送 |
| 结果审核 | 查看证据并纠错 | 根据修正事实重算总结和评分 | 标记已修正项和报告版本 | 用户修正优先于 AI 推断 |
| 改进闭环 | 选择、忽略或完成建议 | 后续检查相关行为是否出现 | 展示建议状态和前后证据 | 不自动操作外部系统 |

## 7. 报告周期与时间窗口

### 7.1 统一规则

- 所有周期使用左闭右开区间，包含开始时刻，不包含结束时刻。
- 时区默认跟随系统时区，每份报告固化记录生成时使用的时区。
- 电脑在计划时点休眠或关机时，下次启动后自动补生成，并标记“延迟生成”。
- 手动重算更新同一时间窗口的报告版本，不创建重复周期。
- 后续增量扫描发现属于已生成窗口的迟到事件时，标记对应报告“数据已更新”，自动创建同一窗口的新版本并重算；事件按稳定来源标识去重，不得重复计数。

### 7.2 生成周期

| 报告 | 时间窗口 | 自动触发 | 手动操作 |
| --- | --- | --- | --- |
| 日报 | 昨日 19:00 至今日 19:00 | 每日 19:00 | 支持指定任意日报窗口生成或重算 |
| 周报 | 上周日 19:00 至本周日 19:00 | 每周日 19:00 | 支持指定周生成或重算 |
| 月报 | 上月 1 日 19:00 至本月 1 日 19:00 | 每月 1 日 19:00 | 支持指定月生成或重算 |

跨越 19:00 的任务可以被多个报告引用，但每个报告只统计本窗口内发生的事件和活跃时间。

## 8. 页面结构与信息布局

### 8.1 导航结构

- **概览**：默认首页，支持日、周、月切换和时间窗口选择。
- **任务与会话**：查看工作区间、任务、项目、证据和纠错入口。
- **成长**：查看 28 天成熟度、五维趋势、风险信号和改进闭环。
- **数据源**：查看五类工具的检测、权限、版本、最近扫描和完整度。
- **设置**：配置模型 API、分析模式、调度、时区、隐私和导出。

### 8.2 概览页信息层级

1. **周期栏**：日/周/月、当前窗口、数据完整度、最后生成时间、手动重算。
2. **数据总览**：活跃时间、会话、消息、Token、工具调用、Skill、Subagent、项目和任务数。
3. **协作水平**：今日得分、置信度、近 28 天 L1-L4、五维分布和趋势。
4. **工作成果**：时间轴、任务名、项目、产出摘要、验证状态和证据。
5. **AI 教练**：最多 3 条建议，每条包含问题证据、建议动作和验证方式。

### 8.3 数据口径

- Token 在可用时分别展示输入、输出、缓存和总量；仅有总量时不伪造分项。
- 会话优先按源工具 session ID 去重；无稳定 ID 时标记为推定会话。
- 活跃时间由消息和工具事件间隔推算，不等同于应用打开时长。
- 并行 Agent 执行时间单独展示，不与人工活跃时间直接相加。
- 项目优先从工作目录和仓库根目录识别；缺失时允许人工归类。

### 8.4 首次扫描与工作区间规则

- 首次扫描在本地索引所有可发现且可解析的历史日志，并显示按数据源划分的进度；不得自动把全部历史内容发送给模型。
- 默认只对当前报告窗口执行标准 AI 分析；历史窗口仅在用户主动生成、重算或发起深度分析时进入对应分析流程。
- 同一会话中，相邻事件时间差不超过 5 分钟时归入同一活跃段；超过 5 分钟则拆分为不同活跃段。
- 同一项目、同一任务的相邻活跃段间隔不超过 20 分钟时可自动合并；超过 20 分钟保留为独立工作区间。
- 不同项目默认不自动合并；缺少项目证据或任务归属冲突时保留为候选并展示置信度，等待用户纠错。
- 并行 Agent 作为独立执行区间记录，可与人工活跃段存在时间重叠，但不得相加为人工活跃时间。

## 9. 评分与 L1-L4 成熟度

### 9.1 双层评分

- **今日协作得分**：0–100，反映当日报告窗口内的协作质量，允许随任务特性波动。
- **近 28 天成熟度**：按活跃日和有效证据滚动计算，反映稳定能力。

近 28 天活跃不足 3 天时，显示“证据不足”，不输出 L1-L4。

### 9.2 五个评分维度

| 维度 | 权重 | 核心问题 | 正向证据示例 |
| --- | ---: | --- | --- |
| 任务定义 | 20% | 目标、约束、上下文和交付标准是否清楚 | 明确输入、边界、验收和不做范围 |
| 协作编排 | 20% | 工具、Skill、Subagent 和拆解是否匹配复杂度 | 合理选用工具与并行，避免无关调用 |
| 迭代深度 | 20% | 是否根据中间结果修正假设和方案 | 有反馈、调整、重试、对比和收敛 |
| 验证闭环 | 25% | 是否用测试、Review 或证据确认结果 | 运行验证、处理失败、检查 diff 和用户可见结果 |
| 资产沉淀 | 15% | 是否形成可复用能力 | Prompt、Context Pack、Workflow、Skill、Loop 或方法文档 |

评分必须同时考虑任务复杂度、行为证据、结果证据和证据置信度。使用量本身不奖励；低 Token 但目标清楚、验证充分的任务不应被惩罚。

### 9.3 聚合与等级门槛

- 每个有效任务分别计算五维分数；证据不足的维度标记“不可评分”，不按 0 分处理。
- 任务复杂度分为简单、中等、复杂，聚合权重分别为 1、2、3；复杂度由任务目标、约束和交付范围判定，不按 Token、时长或工具调用量判定。
- 今日得分是报告窗口内有效任务分数的复杂度加权平均；没有有效任务时显示“证据不足”。
- 近 28 天分数是有效活跃日得分的等权平均，不按某天 Token 或任务数加权；有效活跃不足 3 天时不输出等级。
- L3 除总分达到 60 外，近 28 天必须至少存在 3 个具备迭代证据且完成结果验证的任务。
- L4 除总分达到 80 外，近 28 天必须至少存在 5 个具备迭代和结果验证的任务，并至少有 2 个任务形成可核对的复用资产。
- 未满足证据门槛时，等级降至满足全部条件的最高层级，并明确展示缺失证据。

### 9.4 成熟度定义

| 层级 | 近 28 天得分 | 产品语义 |
| --- | ---: | --- |
| L1 基础使用 | 0–39 | 以一次性问答为主，缺少稳定的任务定义、迭代和验证 |
| L2 结构化执行 | 40–59 | 能给出较清楚任务和上下文，使用工具完成明确交付 |
| L3 迭代式协作 | 60–79 | 能拆解复杂任务，根据结果迭代并形成验证闭环 |
| L4 系统化复用 | 80–100 | 过程可控、可验证，并持续沉淀可复用资产 |

L3 和 L4 除总分门槛外，还需要相应的验证、迭代或资产证据，不允许仅靠使用量、单项高分或单次任务达成。

### 9.5 效率风险信号

“过度思考”不作为独立维度，而作为任务级风险信号：

- 多轮推理没有产生新证据、新决策或新产出。
- 重复读取、重复失败或无收敛的工具循环。
- 上下文或 Token 增长与任务复杂度不匹配。
- 简单任务使用过多 Subagent、Skill 或编排层。
- 宣称完成但缺少可观察的结果验证。

风险信号必须附带证据和置信度，不能仅因 Token 多或时间长触发。

## 10. 标准分析与深度分析

### 10.1 模型接入

- V1 支持 OpenAI-compatible API 和 Anthropic API。
- 用户自行配置模型服务地址、API Key 和模型，并承担相关费用。
- API Key 保存在 macOS 系统钥匙串，不进入报告或导出。
- 无可用模型时仍生成指标报告，语义部分显示“未生成”。

### 10.2 标准模式

标准模式用于自动日、周、月报告。本地程序先生成最小分析包，可包含 Prompt 的任务性摘要或必要片段、关键结论、工具调用摘要、验证结果和统计指标。

默认过滤完整代码、超长工具输出、重复历史、明显密钥和无关二进制内容。

首次启用标准模式时，系统必须展示拟发送的内容类型、目标模型服务、模型、服务方数据保留提示和关闭方式；用户明确同意后才可启用自动语义分析。未授权或撤回授权时只生成指标报告。授权记录保存在本地，模型服务地址或内容范围发生实质变化时需要重新确认。

### 10.3 深度模式

- 只能从指定日期、任务或会话主动发起，不参与默认定时任务。
- 每次发送前展示内容类型、时间范围、预计 Token、目标模型服务和模型。
- 用户确认后才发送；退出或超时都视为取消。
- 深度分析失败不删除标准结果，可修改模型、缩小范围或重试。

## 11. 本地数据与改进闭环

### 11.1 本地事实源

- SQLite 是报告、纠错和趋势的事实源。
- 原始日志保留在各源工具目录；产品保存结构化事件、必要摘要、原始引用和置信度，不建立完整会话副本。
- Markdown、JSON 以及未来日历集成都只是事实库的投影视图，不作为周报或月报的反向汇总源。

### 11.2 用户纠错

用户可以：

- 修改任务名和项目归属。
- 合并或拆分相邻工作区间。
- 排除或恢复会话。
- 补充任务产出和验证状态。
- 对评分证据标记有帮助或判断不准确。

修正保存在本地，报告重算时优先于 AI 推断。

### 11.3 改进建议闭环

每条建议包含问题证据、建议动作和验证信号。用户可设置为“本周实践”“已完成”或“忽略”。后续报告只能依据新会话中的可观察证据提示“可能改善”，不得无证据宣告完成。

### 11.4 本地运行、安全与数据生命周期

- 本地 Web 服务 V1 只绑定 `127.0.0.1`，不监听局域网地址；非本机来源请求不得访问应用数据。
- 后台服务在用户登录后启动，浏览器关闭不影响扫描和 19:00 报告任务；休眠或关机错过任务时在下次服务启动后补生成。
- 结构化事件、分析结论、报告版本和用户修正默认保留到用户主动删除。
- 用户可按时间范围、数据源或全部清除本地产生的数据；清除操作不得删除或修改源工具日志。
- API Key 通过独立操作从 macOS 系统钥匙串清除；导出、日志和错误信息不得包含 Key。

## 12. 产品模块拆解

| 模块 | 产品职责 | 主要输入 | 主要输出 |
| --- | --- | --- | --- |
| 数据源管理 | 发现工具、路径、权限、版本和扫描状态 | 本机环境、用户指定路径 | 数据源状态和扫描计划 |
| 日志适配 | 将五类日志转为统一事件 | 原始日志 | 会话、消息、工具、Token、项目事件 |
| 工作区间与任务 | 推断活跃区间、并行执行、项目和任务 | 统一事件、用户修正 | 工作区间、任务候选、证据引用 |
| 指标引擎 | 计算日、周、月统计 | 统一事件、工作区间 | 使用数据和趋势 |
| AI 分析 | 生成任务、产出、评价和建议 | 标准或深度分析包 | 语义结论、证据、置信度 |
| 评分引擎 | 合并确定性信号和 AI 评价 | 指标、复杂度、语义证据 | 今日得分、五维得分、成熟度 |
| 报告与看板 | 展示、纠错、重算和导出 | 指标、结论、评分、修正 | 日、周、月看板及 Markdown/JSON |

## 13. 状态反馈与人工接管

| 状态 | 用户可见反馈 | 可用操作 |
| --- | --- | --- |
| 扫描中 | 当前数据源、已发现文件和新增事件 | 可离开页面，不重复启动同一扫描 |
| 整理中 | 正在识别工作区间和任务 | 查看已完成的指标数据 |
| AI 分析中 | 目标模型、分析模式和进度 | 可取消手动深度分析 |
| 等待确认 | 深度分析范围、预计 Token 和隐私提示 | 确认、缩小范围或取消 |
| 部分成功 | 已完成数据源和未完成分析 | 查看可用结果、重试失败部分 |
| 生成完成 | 数据完整度、生成时间和报告版本 | 查看、纠错、导出或重算 |
| 生成失败 | 失败阶段、受影响范围和可执行原因 | 重试、修改配置或生成指标报告 |

必须人工确认的情况：

- 首次访问非默认日志目录。
- 每次深度分析发送更完整内容之前。
- 大范围排除、批量重新归类或删除本地派生数据之前。
- 导出包含证据片段而非仅统计和摘要的报告之前。

## 14. 异常与边界

| 情况 | 产品处理 |
| --- | --- |
| 工具未安装或未发现日志 | 显示“未发现”，不影响其他数据源 |
| 日志无权限或格式变化 | 显示具体数据源和可执行授权或升级指引 |
| 只有 Token、无消息内容 | 保留指标，语义任务和评分标记“证据不足” |
| 不同工具 Token 口径不同 | 分工具展示口径，总览标注不完全可比 |
| 多 Agent 并行执行 | 单独计算 Agent 区间，不与人工时间简单累加 |
| AI API 超时、限流或欠费 | 保留指标和已完成结果，支持重试或指标报告 |
| AI 返回不完整 | 仅接受可验证部分，其余标记失败 |
| 用户更换时区 | 旧报告保留原时区；新报告使用新时区 |
| 原始日志删除或移动 | 保留派生报告，原始引用标记不可访问 |
| 已生成窗口出现迟到日志 | 去重导入，标记受影响报告并自动生成同窗口新版本 |
| 首次扫描历史量很大 | 按数据源展示进度并可离开页面；禁止重复启动同一扫描 |
| 标准分析尚未授权或授权撤回 | 只生成指标报告，不向模型服务发送内容 |
| 本地服务被非本机地址访问 | 拒绝请求并且不返回应用数据 |

## 15. 验收标准

### 15.1 数据与周期

- [ ] 五个目标工具均有独立状态和最近扫描结果。
- [ ] 受支持样本重复扫描两次不重复计数。
- [ ] 缺失数据展示不可用原因，不展示为 0。
- [ ] 源日志全程只读。
- [ ] 日报、周报、月报严格使用已定义的 19:00 左闭右开窗口。
- [ ] 错过定时触发后能补生成并标记延迟。
- [ ] 手动重算同一窗口不创建重复报告。
- [ ] 首次扫描可索引全部可解析历史，但只分析用户当前请求的报告窗口，且进度可观察。
- [ ] 相邻事件、工作区间、跨项目和并行 Agent 严格按 5 分钟、20 分钟及不相加规则处理。
- [ ] 迟到事件不会重复计数，并会触发受影响窗口的新报告版本。

### 15.2 看板与复盘

- [ ] 概览页依次展示数据、协作水平、工作成果和教练建议。
- [ ] 用户可切换日、周、月，并查看精确窗口与数据完整度。
- [ ] 任务至少展示任务名、项目、工作区间、产出、验证状态和证据入口。
- [ ] 用户可完成重命名、项目修改、合并、拆分、排除和重算。

### 15.3 评分与可解释性

- [ ] 今日得分和五维分数均可下钻到证据、来源引用和置信度。
- [ ] 使用量增加不会在没有质量证据时直接提高分数。
- [ ] 活跃不足 3 天时不输出虚假 L1-L4。
- [ ] L3/L4 需要验证、迭代或资产证据。
- [ ] 今日得分按任务复杂度 1/2/3 加权，28 天得分按有效活跃日等权聚合。
- [ ] 使用每工具至少 20 个、总计至少 100 个标注任务组验收，整体一致率不低于 80%，单工具不低于 70%。
- [ ] 效率风险信号不因 Token 多或时间长单独触发。

### 15.4 AI 与隐私

- [ ] 未配置 AI API 时仍可生成指标报告。
- [ ] 标准模式不发送完整大型工具输出和明显密钥。
- [ ] 标准模式首次外发前展示内容、模型服务和保留提示，并取得明确授权；撤回后停止自动语义分析。
- [ ] 深度模式每次发送前展示范围、预计 Token、目标模型服务和模型。
- [ ] AI 失败时保留可用指标和已完成结果。
- [ ] 应用不上传匿名遥测、用户指标、报告或原始会话。
- [ ] 本地 Web 服务只绑定 `127.0.0.1`，浏览器关闭后后台调度仍运行。
- [ ] 用户可清除派生数据或钥匙串中的 API Key，且清除派生数据不修改源日志。

### 15.5 导出与开源

- [ ] 日、周、月报告均可导出 Markdown 和 JSON，并标注周期、时区、生成时间和完整度。
- [ ] 导出原始证据片段需要单独确认，默认不包含完整会话。
- [ ] 仓库使用 Apache-2.0，只包含脱敏合成 fixtures，不包含真实日志、API Key 和用户报告。

## 16. 已确认决策

- 独立本地开源产品，不作为 Temporal 内置模块。
- 首期支持 Codex、Claude Code、OpenCode、WorkBuddy 和 Qoder。
- 载体为本地 Web 应用，后续可用 Tauri 包装。
- 首页优先级为数据看板、工作成果、AI 教练。
- 评分采用今日得分与 28 天 L1-L4 双层结构，以及五个能力维度。
- 语义分析使用用户自有 API，支持标准和深度两种模式。
- 报告以每日 19:00 为日边界，周报周日 19:00、月报每月 1 日 19:00 生成。
- V1 使用 Apache-2.0，不启用匿名遥测。

## 17. 待验证假设

以下不是 PRD 决策阻断项，但必须在技术调研和样本验收中验证：

1. OpenCode、WorkBuddy 和 Qoder 的目标版本存在稳定可发现、允许本地解析的会话或活动日志。
2. 五类工具至少 80% 的活跃会话具有足够时间戳和项目证据。
3. 最小分析包仍能支撑任务、产出和五维协作评价。
4. 任务名和项目归属的用户可识别率能够达到 80%。
5. 评分规则可通过脱敏合成样本和用户本地历史会话双路校准；真实样本不进入 Git。
