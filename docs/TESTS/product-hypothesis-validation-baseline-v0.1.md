## 0. 文档信息

| 字段 | 内容 |
| --- | --- |
| 状态 | 初步验证完成，尚未达到产品化门禁 |
| 验证日期 | 2026-07-18 |
| 被验证版本 | `v0.2.2` / commit `50b5d4d` |
| 数据范围 | 2026-07-15 至 2026-07-18 的 4 份本地日报；2026-07-18 窗口内 4 个有用户目标的 Session 结构抽样 |
| 分析模式 | Deterministic；AI consent 当前为 revoked/not granted |

本报告验证的是“当前版本和本地日志能否支撑长期产品目标”的基础假设，不是发布验收，也不是对用户个人工作能力的评价。

## 1. 隐私与证据边界

- 真实 Codex 日志和日报仅在本机只读检查。
- 本文只记录聚合统计、结构特征、脱敏后的问题类型和结论。
- 不写入原始 Prompt、响应、工具输出、项目路径、Session ID 或报告正文。
- 未重新授权 AI analysis，没有把本地证据包发送给模型。
- 真实样本 ground truth 不进入 Git；后续若需要保存，只能使用本地私有标注文件或不可逆脱敏样本。

## 2. 验证方法

本次使用四类证据：

1. 静态检查 `adapter/parser/task/round/evidence/scoring` 实现，确认每层保留和丢弃的字段。
2. 聚合检查最近 4 份已生成 `report.json`，观察 Task、Relation、Evidence Packet、验证状态和标题质量。
3. 对最新窗口内 4 个有真实用户目标的 Session 做结构审计，只统计消息、工具、结果、usage、turn context 和 compaction。
4. 人工抽查最新窗口的任务边界，判断同一 Session 内目标是否被正确合并或拆分。

这是一轮方向性 baseline，不替代后续 20-30 个真实 Task 的正式人工标注评测。

## 4. 假设验证结论

### H1：本地日志包含足够信息重建任务主干

**结论：初步支持。**

原始日志同时存在用户输入、Assistant 响应、工具参数、工具结果、usage、turn context 和 compaction。最新窗口抽样中，每个有用户目标的 Session 都能人工识别其目标和结果主干。

但“原始日志足够”不等于“当前解析结果足够”。日志中有大量系统注入、技能快照、环境上下文和重复事件，需要经过正确归一化才能使用。

### H2：渐进压缩保留了完成判断所需的核心证据

**结论：部分支持，但当前实现存在关键证据损失。**

正向证据：

- 每个当前 Task 都生成了独立 Evidence Packet。
- 26/26 Packet 都保留 outcome，能够提供结果摘要候选。
- Packet 有 Task 独立预算，不再用全日报前 N 条消息代表所有任务。

反向证据：

- Parser 将 Tool Result 归类为事件，但不保留结果正文，因此测试通过/失败、文件变更结果和错误原因可能在 Unified Event 层丢失。
- Assistant 的过程性 commentary 统一被标为 outcome，不能区分计划、进度和最终结果。
- 同一 Assistant 内容可能同时从不同 Codex 事件通道进入，Evidence Packet 中已观察到重复锚点。
- 26/26 Packet 都发生截断；截断本身合理，但当前 coverage 只说明数量，不说明被省略内容的重要性。
- 14/26 Packet 缺少 verification；抽查中存在“最终摘要明确描述已核验”但 Task 仍为 not_observed 的情况。

因此当前主要问题不是 AI 上下文预算不足，而是压缩前的事件语义分类和结果保真度不足。

### H3：多目标会话和跨会话任务可以可靠重建

**结论：尚未支持，已发现明确反例。**

最新窗口报告输出 4 个 Task。人工抽查发现其中 3 个边界基本合理，另 1 个长 Session 至少包含 4 个明显不同的用户目标，但被合并为一个 Task，并出现“标题来自第一个目标、outcome 来自最后一个目标”的错配。

最近 4 份日报中另有 6/26 个标题受到 ambient context、Skill snapshot 或 Selected Text 污染，1 个标题脱敏后只剩路径。说明当前标题归一化和新目标识别仍不稳定。

当前规则主要依赖显式转场词、Issue/PR/文件锚点和时间邻近；对于用户直接提出新问题、没有使用“新任务/接下来”等转场词的长会话，容易发生欠切分。

### H4：可以可靠识别跨 Session 关联

**结论：数据可用，准确率未证明。**

当前日志提供 source session、Subagent run、共同交付物和 continuation 线索；4 份日报中已形成 9 个跨 Session Task。

但 26 个 Task 同时产生 191 条 Relation，重度 Subagent 日期分别出现 90 和 101 条 Relation。关系层目前更像原始关系边集合，还没有通过人工标注证明它们是否应该全部合并、聚合或展示。

### H5：可以从当前日报识别稳定习惯和回答偏好

**结论：当前版本无法验证。**

当前报告按天独立生成，没有跨日报用户模型、显式偏好事件、纠错记录或稳定性窗口。日志中虽然存在用户反馈，但当前只把后续用户消息粗略归为 feedback，无法区分目标扩展、事实纠错、表达偏好和验收意见。

在增加跨天记忆之前，应先定义偏好来源优先级：用户显式声明 > 对具体回答的反复纠正 > 多次稳定行为 > 单次行为。单日行为不得直接写入长期画像。

### H6：可以发现高价值资产候选

**结论：当前版本只能发现弱候选，不能证明价值。**

当前实现能识别文件变更和部分“创建/更新文档、脚本、测试、Skill”等文本，但没有稳定的资产实体、重复模式 ID、适用场景、确认状态和维护成本。Evidence Packet 的 asset 类别还可能被文件名、Skill 快照或 Selected Text 触发。

因此当前的 `hasReusableAsset` 只能表示“观察到可能的资产变更”，不能表示它值得复用，也不能表示已经复用。

### H7：资产能被后续调用并产生改善

**结论：当前版本无法验证。**

产品没有 Asset、Reuse 和 Outcome Change 的持久对象，也没有把后续任务与历史资产调用关联。当前日报最多能提出建议，无法记录建议是否采纳、资产是否调用、为什么未调用以及调用后表现。

这是长期产品价值的核心缺口，但不应立即扩成完整平台。应先用一个人工确认的资产候选做最小闭环实验。

### H8：用户控制和隐私边界足以支撑持续使用

**结论：本地与授权边界初步支持，纠错体验尚未验证。**

当前版本只读本地日志，真实报告位于 Git 之外；AI consent 已撤回，4 份样本中的 26 个 Task 均为 deterministic，没有发生 AI enrichment。说明“不授权时仍能生成事实报告”的边界成立。

但用户目前不能在报告中纠正任务边界、标题、项目、结果或建议，也没有纠错成本与采纳率数据，因此长期信任尚未证明。

## 5. 总体判断

当前证据支持以下判断：

> Codex 原始日志足以成为个人协作改进系统的数据起点，但 `v0.2.2` 还不能稳定把原始信息转化为可信 Task、验证证据、长期偏好和资产复用闭环。

当前最先要解决的不是增加更多数据或让 AI 阅读更多原文，而是提高三层转换质量：

```text
Raw Log
→ 可信 Unified Event
→ 可信 Task / Round / Evidence Packet
→ 才允许生成洞察、资产候选和长期模型
```

## 6. 产品化门禁前的正式验证计划

### 6.1 Ground Truth 样本

从真实日志中私下标注 20-30 个 Task，至少覆盖：

- 单 Session、单目标、少轮次。
- 单 Session、同一目标多次迭代。
- 单 Session、多个不同目标。
- 跨 Session continuation。
- 主任务与 Subagent delegation。
- 有明确验证通过/失败的任务。
- 形成文档、脚本、测试、Workflow 或 Skill 候选的任务。

每个样本人工标注：目标、边界、项目、关键轮次、最终结果、验证状态、资产候选和跨 Session 关系。

### 6.2 建议门槛

| 维度 | 进入下一阶段的建议门槛 |
| --- | ---: |
| Task 边界整体一致率 | >= 80% |
| 多目标 Session 切分召回率 | >= 80% |
| 项目归属一致率 | >= 80% |
| Intent/Outcome 主干保留率 | >= 90% |
| 明确验证结果召回率 | >= 80% |
| 注入上下文成为标题 | 0 个 |
| 跨 Session 合并 precision | >= 80% |
| Evidence Packet 核心结论可回溯率 | 100% |

### 6.3 一个最小资产闭环实验

在不建设完整资产平台的前提下：

1. 从至少 3 次相似协作中人工确认一个重复摩擦。
2. 选择最小资产类型，不默认使用 Skill；可以是 Prompt、规则、Checklist、Context Pack 或 Workflow。
3. 在下一次相似任务中明确调用该资产。
4. 记录是否减少重复解释、无效轮次或返工，是否提高验证完整度。
5. 用户判断资产是保留、修改还是废弃。

只有这个最小闭环产生正向证据，才进入长期资产模型和产品化设计。

## 7. 当前决策

1. 保留 `v0.2.2` 作为事实采集和日报实验载体。
2. 暂停因长期愿景而扩展 Dashboard、多工具、周月报和自动 Skill 写入。
3. 下一阶段命名为“数据与任务重建假设验证”，不是“Dashboard v0.3”。
4. 先完成正式 ground truth 评测和一个最小资产闭环，再决定下一版 PRD。
5. 当前 AI enrichment 未授权，因此不能用本次结果判断 AI 洞察质量；若要验证，必须单独明确授权并限定脱敏样本。

## 8. 局限

- 本次人工深查集中在最新窗口，不代表所有历史日志格式。
- 聚合统计来自 4 份日报，其中重度 Subagent 日对 Relation 数量影响很大。
- 没有保存真实 ground truth 标注，因此本报告给出的是 baseline 和明确反例，不是最终准确率报告。
- AI enrichment 未运行，不能验证任务改名、Session Insight 和 AI Coach 的真实质量。
