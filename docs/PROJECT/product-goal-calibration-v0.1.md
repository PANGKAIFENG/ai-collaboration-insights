# AI Collaboration Insights 产品目标校准 v0.1

## 0. 文档信息

| 字段 | 内容 |
| --- | --- |
| 状态 | 已对齐，作为后续发现与验证工作的目标基线 |
| 日期 | 2026-07-18 |
| 当前产品版本 | `v0.2.2` |
| 当前验证范围 | Codex-only、本地日报 |
| 适用阶段 | 产品化设计之前的目标与假设验证 |

本文不替代当前 MVP PRD，也不授权扩展 Dashboard、多数据源、周月报或自动资产写入。它回答的是更上层的问题：产品长期为什么存在、最终要产生什么结果，以及在进入下一轮产品化前必须先证明什么。

## 1. 校准结论

AI Collaboration Insights 的长期方向合理、必要且具备分阶段实现的可能，但“日报、Dashboard、会话分析、偏好识别、Skill 沉淀”不能同时被当作产品目标。它们分别是入口、界面、分析能力、模型能力和资产类型。

产品真正需要证明的不是“能够生成更多分析”，而是：

> 过去的协作经验，能够让下一次相似任务完成得更好。

“更好”至少包括更少重复解释、更少无效轮次和返工、更高交付质量、更完整验证，以及更稳定地调用正确的协作资产。

## 2. 三层目标体系

### 2.1 长期愿景

> 让个人与 AI 的协作不再每次从零开始，而是形成一个持续学习、持续复用、持续改善的工作系统。

### 2.2 产品使命

> 基于本地真实协作记录，识别有效模式与重复问题，把它们转化为可验证、可复用、可持续优化的个人协作资产。

### 2.3 核心用户结果

> 历史协作资产在后续相似任务中被正确调用，并带来可验证的质量、可靠性或效率改善。

优先关系如下：

1. 最终结果：提升协作质量与可靠性。
2. 伴随收益：减少重复解释、无效轮次、返工和时间消耗。
3. 实现机制：积累并持续优化个人协作资产。

## 3. 产品定位

更准确的长期定位是“个人 AI 协作改进系统”，而不是“AI 使用数据看板”。

| 层级 | 需要回答的问题 | 典型产物 |
| --- | --- | --- |
| 观察层 | 发生了什么 | 会话、任务、轮次、工具、Token、时间和结果事实 |
| 理解层 | 为什么顺利或受阻 | 重复模式、偏好、摩擦、验证缺口和协作策略 |
| 改进层 | 下一次应该改变什么 | 建议、Prompt、Context Pack、Workflow、规则、项目知识或 Skill 候选 |
| 复用层 | 改进是否真的被使用 | 资产检索、调用记录、适用场景和版本 |
| 验证层 | 使用后是否变好 | 质量、返工、验证、耗时和用户反馈变化 |

日报是观察入口，Dashboard 是可能的交互界面，Skill 只是资产类型之一。任何单一形态都不能代表完整产品价值。

## 4. 核心闭环与 Crux

长期闭环为：

```text
协作事实
→ 任务与结果重建
→ 重复模式或摩擦识别
→ 改进/资产候选
→ 用户确认
→ 后续任务检索与调用
→ 效果验证
→ 资产和个人协作模型持续修正
```

最关键的 Crux 是稳定建立以下可追溯关系：

```text
Task → Pattern → Asset → Reuse → Outcome Change
```

如果只能做到 `Task → Pattern → Suggestion`，产品仍然是一份分析报告；只有继续证明真实复用和结果变化，才形成改进系统。

## 5. 问题分类与领域定位

这不是单一的功能设计问题，而是以“发现问题”为主、兼有数据治理和评测设计的问题。

| 领域 | 可借鉴的成熟概念 | 对本产品的意义 |
| --- | --- | --- |
| Personal Analytics / Quantified Self | 行为事实、趋势和自我反馈 | 建立可信事实层，但不能把使用量等同于能力 |
| Personal Knowledge Management | 捕获、组织、检索和复用 | 管理项目知识、偏好和上下文资产 |
| Continuous Improvement | 观察、干预、验证、改进 | 要求建议进入后续实践并验证效果 |
| Human-AI Co-adaptation | 人与 AI 双向适应 | 用户修正系统，系统逐步适应用户 |
| Learning Analytics | 证据、能力模型和反馈 | L1-L4 必须有证据和稳定性，而不是单日人格评分 |

## 6. 可实现性分层

| 能力 | 当前判断 | 主要难点 |
| --- | --- | --- |
| 会话、轮次、Token、工具和时间事实 | 高可行 | 事件口径、去重和窗口归因 |
| 任务、项目和结果重建 | 中高可行 | 多目标会话、跨会话延续、系统注入污染 |
| 协作习惯与问题识别 | 中等可行 | 需要跨天样本、置信度和用户纠错 |
| 回答偏好建模 | 中等可行 | 显式偏好与从行为猜测必须分开 |
| 资产候选发现 | 中等可行 | 重复不等于值得资产化，需判断复用价值 |
| 自动生成高质量资产 | 中低可行 | 需要人工确认、测试、版本和生命周期治理 |
| 证明资产带来改善 | 最难但必要 | 需要相似任务基线、真实调用和结果度量 |

## 7. 必须验证的假设

| ID | 假设 | 如果错误会怎样 | 当前可验证方式 |
| --- | --- | --- | --- |
| H1 | Codex 本地日志包含足够信息重建任务主干 | 产品只能做用量统计，无法理解工作 | 对真实日志抽样核对目标、约束、尝试、结果和验证 |
| H2 | 渐进压缩后仍保留完成判断所需的核心证据 | AI 获得的是失真摘要，洞察和评分不可信 | 比较原始日志、Unified Event、Evidence Packet 三层信息损失 |
| H3 | 多目标会话和跨会话任务可以可靠切分、关联 | 成果、耗时、评分和建议都会错配 | 建立人工 Task ground truth，测边界 precision/recall |
| H4 | 可以从跨天行为中识别稳定习惯和偏好 | 把偶然行为误判成人格或长期习惯 | 至少跨 2-4 周，区分显式偏好、重复行为和单次事件 |
| H5 | 重复模式可以转化为高价值资产候选 | 产生大量低质量 Skill 和维护负担 | 人工评价候选的重复性、复用范围、收益和维护成本 |
| H6 | 资产能在后续任务中被正确检索和调用 | 资产只被存储，不产生实际价值 | 记录候选、确认、版本、调用任务和未调用原因 |
| H7 | 使用资产后的改善可以被验证 | 产品无法证明长期价值 | 对相似任务比较返工、轮次、验证和用户评价 |
| H8 | 用户愿意持续纠错并信任本地分析 | 模型长期漂移，用户逐渐不再查看 | 观察纠错成本、采纳率、拒绝原因和留存 |

## 8. 北极星指标

长期候选北极星指标：

> 经验证的协作改善率：使用历史沉淀资产后，在相似任务中获得可验证改善的任务比例。

辅助指标包括：

- 历史资产真实复用率。
- 重复上下文减少量。
- 平均返工和无效轮次变化。
- 验证闭环率变化。
- 用户接受、修改和拒绝资产候选的比例。
- 错误任务重建、错误建议和错误资产候选率。

Token、会话数、消息数、Subagent 数和 L1-L4 等级只作为事实或解释维度，不作为北极星指标。

## 9. 当前阶段门禁

进入下一轮产品化设计前，当前 Codex-only 版本至少需要证明：

1. 日志能够支撑可信的 Task 主干，而不只是碎片消息。
2. 多目标会话、跨会话关系、结果和验证证据达到可接受准确率。
3. 当前 Evidence Packet 没有系统性丢掉结论所需证据。
4. 至少能识别一个真实重复问题，并定义可验证的资产候选。
5. 产品能够记录该候选后续是否被使用；如果当前版本不能，必须明确这是下一阶段的最小能力，而不是继续美化日报。

## 10. 不建议现在做

- 不因为长期愿景扩大而立即接入更多 AI 工具。
- 不先建设动态 Dashboard、SQLite 或长期服务。
- 不继续精修 L1-L4，直到任务、验证和资产证据经过真实样本校准。
- 不自动创建或修改 Skill、Prompt、Workflow 和项目知识。
- 不用更多 Token、更长会话或更多 Subagent 推断协作质量。
- 不在没有真实复用记录时宣称“持续学习”或“长期改善”。

## 11. 可复用的目标校准方法

以后评估相似产品方向时，按以下顺序执行：

- [ ] 先写长期愿景，不写功能。
- [ ] 明确产品使命和唯一核心用户结果。
- [ ] 把最终结果、伴随收益、实现机制和产品形态分开。
- [ ] 识别真正的价值闭环及最难成立的一段。
- [ ] 按事实、推断、行动、复用、验证分层评估可行性。
- [ ] 为每个关键假设写出错误后果和证伪方法。
- [ ] 用当前版本和真实数据先验证基础假设。
- [ ] 只有通过阶段门禁后，才写下一版本 PRD、架构和开发计划。

可复用开场 Prompt：

```text
不要先设计功能。请先区分长期愿景、产品使命、核心用户结果、实现机制和产品形态；
再指出这个方向必须成立的 5-8 个隐含假设、每个假设错误时的崩溃路径，以及如何用当前版本和真实数据证伪。
最后给出进入产品化设计前的阶段门禁和明确不做项。
```

需要持续警惕的反模式：把专业完整的 PRD 当作方向已经正确；把当前日报路径当作未来产品必然形态；把“日志里存在信息”误写成“当前分析管线已经能够正确使用信息”。
